### Critical Bug Fix: Recruiter Matching Engine (late evening)

**Root cause found and fixed.** `get_resume_from_dilly_profile()` in `dilly_profile_txt.py` had a regex (`\[[\w_]+\]`) that prematurely terminated at resume sub-section headers like `[EDUCATION]`, `[SKILLS]`, `[PROJECTS]` ‚Äî treating them as top-level profile sections. This returned only 130‚Äì210 chars of contact info instead of 2,000‚Äì7,000 chars of full resume content for **26 out of 30 candidates**.

Consequences before fix:
- Tag extraction produced 2‚Äì8 tags (from audit text only) instead of 10‚Äì20 tags (from actual resume skills/experience)
- Cole Poirier ("currently learning ML") ranked above Dilan ("Engineered an AI-driven predictive engine") for ML roles because Dilan's Python/SQL/Data Analysis tags were missing entirely
- Evidence-weighted scoring couldn't differentiate candidates because the evidence snippets came from sparse audit data, not rich resume content

Fixes applied:
1. `dilly_profile_txt.py`: Changed regex to only terminate at known top-level sections (`AUDIT`, `GOALS`, `TRANSCRIPT`, etc.), not resume sub-sections
2. `candidate_index.py`: Added 300-char minimum length check before accepting `load_parsed_resume_for_voice` output ‚Äî prevents short contact-info stubs from blocking the `resume_edited.json` fallback

Impact: Dilan gains +7 tags (6‚Üí13), including Python, JavaScript, SQL, Data analysis. Score simulation shows Dilan beats Cole by 20.1 points for ML role (62.8 vs 42.7). **User needs to run `reindex_all_candidates.py` to rebuild all indexes.**

### Matching Accuracy Improvements (3 changes, same session)

1. **Expanded evidence word lists** (`recruiter_search.py`): Added ~30 strong verbs (spearheading, architecting, facilitating, etc.) and ~10 weak phrases (studying, enrolled, workshop, etc.). Switched from substring matching to word-boundary regex ‚Äî fixes false positive where "led" matched inside "enrolled".

2. **Skill inference chains** (`skill_tags.py`): Deterministic rules that add "obvious but unlisted" tags. E.g., strong ML evidence ‚Üí infer Python + Statistics. Financial modeling ‚Üí Excel. React/Node.js ‚Üí JavaScript. Zero API cost, runs at index time.

3. **LLM-based tag extraction** (`skill_tags.py` + `candidate_index.py`): New `llm_extract_skill_tags()` function replaces regex scanning at index time. LLM reads full resume + ontology, returns tags with confidence (high/medium/low) and evidence quotes. Confidence maps to quality scores: high=1.0, medium=0.65, low=0.30. Regex extraction stays as fallback when API key missing. Cost: ~$0.002/candidate.

**User needs to reindex:** `python3 projects/meridian/scripts/reindex_all_candidates.py` then `python3 projects/meridian/scripts/recruiter_search_eval.py`.
