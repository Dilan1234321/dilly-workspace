# What’s in the app

**Purpose:** Single place to see what’s implemented so you can track what’s in the app vs what’s not.  
**Update this file whenever you add or remove features.**

- **Pull to refresh (mobile touch):** From the top of the scroll position (`GlobalPullToRefresh` in root `Providers`), pulling down shows the **Dilly logo** under the safe area; release after ≥ ~80px triggers reload. While the reload is pending (~420ms before navigation), a **gold elliptical stroke** spins around the logo (oval hugging the wordmark). Then **`window.location.reload()`** on any route.

## Onboarding (pre–main app)

**Premium 8-screen flow (March 2026)** — Standalone routes under `/onboarding/*` (no bottom nav). Layout uses the same **375px** centered column as the rest of the app (`onboarding-app-frame` in `onboarding/layout.tsx`). State persisted in `sessionStorage` (`dilly_onboarding_state_v1`) via `OnboardingProvider` in root `Providers`.

| # | Route | Notes |
|---|--------|--------|
| 1 | `/onboarding/welcome` | Hook: grid + gold glow, .edu + UTampa domain, `POST /auth/send-verification-code`. If a **valid session** already exists (`GET /auth/me`), redirects to **`/`**. |
| 2 | `/onboarding/verify` | 6-digit code, `POST /auth/verify-code`, dev code hint when API returns `dev_code`. After login, **`GET /profile`** — if the student is **already onboarded** (`onboarding_complete`, or name+major(s), or `first_audit_snapshot`), **`router.replace("/")`** instead of continuing to profile. |
| 3 | `/onboarding/profile` | Name, **multiple majors/minors** (UTampa autocomplete, add/remove rows), application target; inferred track + **preProfessional** for Pre-Health / Pre-Law; `PATCH /profile` (same fields as legacy onboarding) |
| 4 | `/onboarding/you-are-in` | Benchmark chart, pills, green CTA → anticipation |
| 5 | `/onboarding/anticipation` | Goldman copy, numbered promises, gold CTA → upload |
| 6 | `/onboarding/upload` | PDF/DOCX only → scanning |
| 7 | `/onboarding/scanning` | Dilly orb + step list + `POST /audit/v2` (min ~11.6s animation before navigate) |
| 8 | `/onboarding/results` | Score reveal, gap callout, Dilly tease, notification pre-prompt (~1.5s), `PATCH` `onboarding_complete`, `/` |

The **home route** (`/`) shows a loading shell until auth + profile resolve, then either redirects to **`/onboarding/*`** (resume via `getResumeOnboardingPath()` + `sessionStorage`) or allows the main app when `onboarding_complete`, legacy profile (name + major), or audit history / `first_audit_snapshot` indicates a returning user. Legacy inline onboarding (magic link, school theme, steps 4–6) was **removed** from `page.tsx`.

Profile saved via PATCH `/profile` (including `onboarding_complete`). Loaded on main app via GET `/profile`.

### Leaderboard (`/leaderboard`)

- **Route:** **`/leaderboard`** (optional **`?track=`**; falls back to cached score-page track or Humanities). Full-bleed dark shell (`.leaderboard-page-root`), Inter, tokens aligned with score page. **Participation:** new profiles default **`leaderboard_opt_in: true`**; missing/`null` counts as in; only explicit **`leaderboard_opt_in: false`** (PATCH `/profile`) opts out. **`projects/dilly/scripts/backfill_leaderboard_opt_in.py`** sets `true` on existing profiles on disk. **GET `/leaderboard-dashboard?track=…`** (aliases **`/leaderboard/page?track=`** / **`/leaderboard/page/{track}`**; `require_auth`, optional **`refresh=true`**) returns podium, ranked entries (free tier: 5-row window + `locked_count`; paid: full list), `move_up_insight`, `weekly_events`, `rank_change`, etc. The app uses **`/leaderboard-dashboard`** so the API URL has no literal `page` segment (avoids bad `track=page` / proxy confusion); junk track strings server-fallback to profile or Humanities. **Pool:** each user is ranked using the **newest audit whose cohort matches** the board (via `get_track_category`), not only their latest run — so a recent audit in another track does not remove them from their usual board. Client cache: `sessionStorage` `dilly_leaderboard_cache_v1` keyed by ISO week + track; bust with `?refresh=true` or **`DILLY_LEADERBOARD_REFRESH_KEY`** after a new audit. **All cohorts:** segmented control **My track** / **All cohorts**; **GET `/leaderboard-dashboard/global`** (alias **`GET /leaderboard/page/global`**, static path before **`/leaderboard/page/{track}`**; client tries both on 404) (`require_auth`, optional **`refresh=true`**) returns the **same JSON shape** as the track board with **`track`: `All cohorts`**: opted-in students ranked by final score on **their profile cohort** (newest matching audit); **list shows top 100**; each **`entries`** / **`podium`** row includes **`cohort_track`**. **`sessionStorage` `dilly_global_leaderboard_cache_v1`** (ISO week). **Voice:** `/voice?context=leaderboard&target_rank=&gap=&weakest_dim=&track=` → **`DILLY_LEADERBOARD_VOICE_PROMPT_KEY`** on home overlay. **Unlock CTA:** `dilly_paywall_source=leaderboard_unlock` + `/?source=leaderboard_unlock`. **Shell:** Bottom nav **Rank** active (no back). **Entry:** main app bottom nav **Rank**; **`/score`** bottom pill **Rank**; peer preview **See full leaderboard**; Insights **Your rank** card **Weekly leaderboard →**; other shells (ATS, Career Hub, certifications, playbook, audit report) include **Rank** tab → `/leaderboard` (with track when known).

### Meridian Voice tab

- **Gemini-style overlay** — When user taps **Dilly AI** in the bottom nav, a floating pill appears at the bottom (like Google Gemini on Samsung). That tap **resumes the most recently updated Voice thread** (does not start a new chat). If there are no saved threads yet, the overlay opens empty until the user sends a message. Empty overlay center copy matches the **Voice empty-state greeting** (first-time long intro vs “Hey {name}…”). Pill expands to show chat + input; "Open full chat" goes to full Voice tab. Overlay floats over whatever screen the user is on.
- Chat UI: message list, **Prompt box** (ai-prompt-box style) for “Ask Meridian…”, rounded-3xl container, auto-resize textarea, send button, bullet rewriter toggle as left action. Enter to send.
- **No auto first bubble** — Opening Voice on an empty chat does **not** inject a proactive “quick insight” (deadline / trajectory / top rec). Empty thread + starter / follow-up suggestion chips only (onboarding first message still comes from `/voice/onboarding-state` when applicable).
- **Typos vs safety** — Inappropriate filter in `voice_helpers.py` applies **Let's keep this professional.** only to clear boundary violations (harassment, hate, threats, etc.), not to typos, glued punctuation (e.g. do.i), or common career misspellings. Charitable decoding is required; `_message_looks_typo_prone` adds a decoding hint on the user turn when needed.
- **Score answers + inline visuals** — When the user asks about Smart/Grit/Build, Dilly can emit `[[scores_visual]]` on its own line; the client strips it and shows **VoiceInlineScoresVisual** (mini radar + three score tiles + overall, optional **vs last audit** deltas, staged “build” CSS animations). ## Backend / API

- **Auth:** POST `/auth/send-verification-code`, POST `/auth/verify-code`; dev_code when MERIDIAN_DEV=1. Session token in `meridian_auth_token`.
- **Profile:** GET `/profile`, PATCH `/profile` (name, major, majors, minors, preProfessional, track, goals, etc.). POST `/profile/photo` (upload), GET `/profile/photo` (serve), DELETE `/profile/photo` (remove).

- **Certifications (optional):** GET `/certifications?uid=…` — when implemented, returns curated list + `dilly_commentary`, `dilly_top_pick_reason`, per-cert `why_it_matters` (cached per audit). Dashboard falls back to client-side build if the route is absent.
- **Explain delta:** POST `/audit/explain-delta` (previous + current audit → why scores changed).
- **Voice:** POST `/voice/chat` (message + optional context → reply). Requires subscribed user.
- **Report:** POST `/report/pdf` → signed URL; GET `/report/pdf/{token}` for download.

Subscribers. Dashboard tries `/audit/career-playbook` first, then `/career-playbook` on 404; if the **final** response is still 404, dashboard builds a client-side brief playbook (no dead end). Powers **`/career-playbook`** page.
- **Ready check:** POST `/ready-check` (target company/role + audit → verdict, summary, gaps). Used by Am I Ready? in Insights.
- **ATS check:** POST `/ats-check` (job_description + audit → missing keywords, suggestions, ready). Used by ATS keyword check in Career Center (Keywords tab).
- **ATS analysis:** POST `/ats-analysis` (file upload → full parseability, extraction simulation, formatting checklist, section completeness, date consistency, ATS readiness score). POST `/ats-analysis-from-audit` (text-based, no upload). Used by ATS Readiness section in Career Center.
- **ATS keyword density:** POST `/ats-keyword-density` (parsed_text + optional job_description → keyword placement map, density score, JD match). Used by Keywords tab in Career Center ATS block.

Body may include `use_llm` (default `true` for API callers). **Full ATS scan** sends `use_llm: false` for fast rule-based fixes; vendor sim receives `ats_analysis` from the first analysis call so the server does not run `run_ats_analysis` twice.

- **Apply through Meridian:** POST `/apply-through-meridian` (body: `job_id`, optional `note`). ## Recruiter API + UI

- **Consent:** All students are treated as consenting to recruiter visibility by joining the app.
- **Backfill:** `python3 projects/meridian/scripts/backfill_candidate_index.py` (optionally `--force`, `--limit N`, `--dry-run`). Requires `OPENAI_API_KEY`. Indexes every profile that has at least one audit.
- **POST /recruiter/search** — Recruiter API key required (`RECRUITER_API_KEY` env; send via `X-Recruiter-API-Key` or `Authorization: Bearer <key>`). Body: `role_description`, `filters` (major, school_id, cities, track, min_smart, min_grit, min_build), `required_skills`, `sort`, `limit`, `offset`. Returns ranked candidates with match_score, semantic_score, skill_fit_score, meridian_fit_score.
- **POST /recruiter/jd-fit** — Same auth. Body: `job_description` (required), `job_title` (optional). Returns Meridian-fit for the role: `smart_min`, `grit_min`, `build_min`, `min_final_score`, `track`, `signals`, `unavailable`. Used to pre-fill min filters when searching (JD → Meridian-fit).
- **GET /recruiter/candidates/:id** — Candidate detail by 16-char profile uid (from search result `candidate_id`). Same auth.
- **Recruiter UI** — `/recruiter`: API key entry (stored in localStorage), then search screen. Describe the role: optional job title, role description textarea, "Get Meridian-fit" (calls jd-fit; shows Smart/Grit/Build bars + track + signals; "Use as filters" fills min_smart, min_grit, min_build, track). Filters: major, track, school_id, cities, min Smart/Grit/Build, required skills, sort. “Find candidates” runs search; results list shows match %, name, major, track, school, scores, “View profile” link. `/recruiter/candidates/[id]`: candidate detail (name, major, track, school, Smart/Grit/Build/final, meridian_take, application_target, job_locations, email). Desktop-friendly; uses existing Meridian design tokens.
- **Recruiter bookmarks & collections (March 2026)** — Bookmark candidates (☆/★) from grid, table, or profile. Create named collections; add candidates via “+” button (modal: pick collection or create new). Right sidebar: general bookmarks + collections; candidate names link to profiles; remove via ×. API: GET/POST/DELETE /recruiter/bookmarks, POST /recruiter/collections, POST /recruiter/collections/add, POST /recruiter/collections/remove, DELETE /recruiter/collections.
- **Recruiter profile enhancements (March 2026)** — **JD gap analysis:** "Strong on X; weak on Y" summary on candidate profile (from jd_evidence_map). **Similar candidates:** "Others like this" section with embedding + score similarity. **Export to ATS:** CSV download of shortlist (name, email, profile link, fit summary). **Recruiter notes:** Private notes per candidate; persists across sessions. API: GET/PUT /recruiter/candidates/:id/notes, GET /recruiter/candidates/:id/similar, GET /recruiter/export/shortlist.
- **Meridian Voice Search (March 2026)** — Conversational candidate discovery on recruiter search page. Floating Voice FAB; describe what you need in plain English ("Find me 5 PM candidates who have shipped production code"); get ranked candidates with evidence in ~30 seconds. Results as inline candidate cards (name, match %, S/G/B, evidence summary, profile link). Multi-turn refinement: "Narrow to CS majors" refines prior search. API: POST /recruiter/voice/search (query, role_description, conversation_history).
- **Matching engine uses Meridian profiles** — Search ranks candidates using an embedding and skill tags built from the full Meridian profile (goals, career_goal, track, locations, etc.), optional “what they told Meridian” text (e.g. from Voice), resume, and latest audit. Profile-only updates (PATCH /profile) trigger re-index so recruiter matching reflects student-stated info even when the resume hasn’t changed.
